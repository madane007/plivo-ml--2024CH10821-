The problem, and where the points were

Corpus: 5.70M chars / 7.32M bytes — 85.8% English, 14.1% Hindi (Devanagari), 0.1% other. In UTF-8 every Devanagari character is 3 bytes, so the stock byte tokeniser spends 3 model steps per Hindi character and, at a fixed steps×batch×block budget, sees the data only ~once. That made the tokeniser the single biggest lever.

What we changed (vs. the deliberately-mediocre baseline)

Tokeniser
  • byte (vocab 256) → trained byte-level BPE, vocab 4096
  • lossless w/ byte fallback (grader round-trip passes)
  • ~3.4 bytes/token → ~3.4× shorter sequences, ~4 effective epochs
Architecture
  • learned pos → RoPE; LayerNorm → RMSNorm
  • GELU MLP → SwiGLU; weight tying on
  • init 0.02 + residual scaled 1/√(2L)
Optimisation
  • Adam const 3e-4 → AdamW, wd 0.1, grad-clip 1.0
  • warmup 100 → cosine, peak 3e-3
  • batch 8 → 64 (more tokens per capped step)
Free readout win
  • Weight EMA (decay 0.999), no extra steps
  • scored raw-vs-EMA in one run; EMA won by −0.025
  • the legitimate "improve after training" trick

What we tried — wins and dead-ends (and why)

Worked — and why
  • BPE tokeniser: the corpus is byte-inefficient (Devanagari = 3 bytes/char). Merging bytes cut sequences ~3.4× → the model both amortises more bytes per prediction and sees ~4 epochs instead of ~1. Biggest single drop: 2.37 → 1.68.
  • Modern recipe (AdamW, cosine+warmup, wd, clip, 1/√(2L) init): the baseline's constant-LR Adam badly under-optimised the 2000 steps; a schedule + higher peak LR use every step.
  • Batch 32→64: steps are capped, tokens are not — more tokens per step directly feeds the data-limited regime (−0.014).
  • Weight EMA: averages out high-LR update noise for free, no extra steps. Best free lever (−0.025). This is the correct answer to "improve after training", not RLHF (which optimises preferences, not likelihood, and would raise bpb).
Dead-ends — and what they taught us
  • Higher peak LR (3e-3 → 6e-3): the LR range-test suggested it, but it gave worse final loss. Lesson: the range test finds the fastest short-term descent, not the best full-schedule peak. Kept 3e-3.
  • Scaling up (vocab 8192 + width E192, ~1.4× params): moved bpb by only 0.008. This was the pivotal negative result — it proved the model is data/step-limited, not capacity-limited, and redirected all further effort to throughput + EMA instead of size.
  • Attention tweak (QK-norm) in the final: a low-risk change to the attention mechanism, folded into the budget-maxed final run — kept only if it beat the saved floor. [result filled in after the final run]

Why we believe this is the right answer

  • Attribution, not luck: gains were isolated — recipe+tokeniser (R1), LR (R2, a negative result we kept), scaling (R3, showed little), and throughput+EMA (R4).
  • The key diagnosis: after the tokeniser fix the model is data/step-limited, not capacity-limited — bigger vocab and bigger network barely moved bpb, so we spent effort on batch throughput and EMA, which did move it.
  • Caps respected honestly: ≤2000 steps; ≤2M non-embedding params (tokeniser/embedding exempt per staff clarification), kept unambiguous by weight tying.
